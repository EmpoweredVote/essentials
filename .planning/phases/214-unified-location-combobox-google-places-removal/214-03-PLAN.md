<objective>
Replace the Results header's Address/Browse mode toggle, address `<input>`, `<LocalityMatches>`, and `<LocationBrowser>` tree with a single `<LocationCombobox>` (SRCH-01), and build the coordinate render path (SRCH-05) — the one genuinely-undecided design point in this phase. The coordinate path is factored into ONE shared internal `resolveCoordinate(lat, lng, raw)` function that BOTH the in-page combobox `onSubmitCoordinate` callback AND an on-mount reader of the `lat`/`lng`/`coord_raw` URL params (the Plan 02 `coordinateRoute` hand-off contract from Landing) invoke — no duplicated logic. It reuses Results' existing `browseResults` direct-injection state (the closest precedent for "inject a list, bypass the address-geocode hook"), synthesizes the D-05 client-side resting label, and extends the `representingCity` banner-hijack guard for the new dispatch.

Purpose: Turns the combobox into a working Results experience across all three input types with no manual mode switch, and makes the Landing→Results coordinate hand-off a concrete, testable mechanism (not a same-wave soft dependency).
Output: Edited `src/pages/Results.jsx` (import/usage of the Google files removed here; the standalone files are deleted in Plan 05).
</objective>
