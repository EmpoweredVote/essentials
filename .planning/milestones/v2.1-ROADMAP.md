# Milestone v2.1: Claude Candidate Discovery

**Status:** ✅ SHIPPED 2026-04-25
**Phases:** 5-7
**Total Plans:** 9

## Overview

Replace manual candidate seeding with a Claude-powered discovery pipeline that scales to any jurisdiction by adding a single config row. Covers the full stack: DB schema, agent runner, orchestration service, admin review UI, email notifications, and weekly cron automation.

## Phases

### Phase 5: DB Foundation + Agent Core

**Goal**: A discovery agent can find candidates from official election authority sources, normalize names, score confidence, detect withdrawals, and persist results to a staging table — all triggerable on demand from the command line.

**Depends on**: Phases 1-4 (v2.0 complete)
**Plans**: 4 plans

Plans:

- [x] 05-01: DB migrations — discovery_jurisdictions, candidate_staging, discovery_runs tables
- [x] 05-02: discoveryAgentRunner.ts — Anthropic SDK wrapper with citation-required output schema
- [x] 05-03: discoveryService.ts — orchestration, name normalization, confidence scoring, withdrawal diff, staging writes, run log
- [x] 05-04: SCHED-01 + STAG-04 + STAG-05 — on-demand trigger endpoint + approve/dismiss API endpoints

**Details:**

- Migration 070 adds three essentials-schema tables: discovery_jurisdictions (jurisdiction registry), discovery_runs (per-run audit log), candidate_staging (pending queue with citation_url NOT NULL)
- Agent runner uses forced `tool_choice=report_candidates` to enforce structured output — 'any' or 'auto' lets Claude pick web_search as final call, producing no typed results
- Discovery orchestrator: load config → call agent → normalize names → score confidence → diff withdrawals → write staging rows → log run
- Four exported pure helpers: normalizeName (lowercase + suffix strip + punctuation removal), nameSimilarity (1 - lev/maxLen), scoreConfidence (domain + fuzzy signals), isDomainAllowlisted (subdomain-aware)
- Withdrawal diff scoped to races where agent returned ≥1 candidate — untouched races produce no false withdrawal flags
- Discovery router mounted BEFORE adminRouter in index.ts — adminRouter's JWT middleware would intercept token-auth requests at the same /api/admin path

### Phase 6: Admin Review UI + Email + Per-Race Trigger

**Goal**: Admin can review, approve, and dismiss staged candidates through a browser UI, receives email when items need attention, and can trigger discovery for individual races.

**Depends on**: Phase 5 (working agent with staging rows to review)
**Plans**: 3 plans

Plans:

- [x] 06-01: Admin staging review UI — extends UnresolvedQueue.jsx pattern; per-item race context, source URL link, approve/dismiss actions
- [x] 06-02: Email notifications — Resend via existing emailService.ts; urgency-aware subject; zero-candidate regression alert
- [x] 06-03: SCHED-02 — per-race on-demand trigger endpoint

**Details:**

- New JWT-gated stagingQueueAdmin.ts router (GET list, POST approve, POST dismiss) leaves Phase 5 X-Admin-Token endpoints untouched — dual-router pattern
- Auth bug discovered and fixed (f474d2a): mount-level requireAuth intercepted X-Admin-Token requests to sibling router; moved to route-level
- StagingQueue.jsx: race grouping via Map keyed on race_id, LEFT JOIN so NULL race_id rows still appear, color-coded confidence badges, 30-day urgency highlighting, optimistic approve/dismiss with 5-second Undo toast
- Email triggers: review notification (urgency-aware subject using uncertainStaged count), zero-candidate regression alert (compares to prior completed run), failure alert (in catch block after run marked failed)
- Per-race trigger uses CTE + LEFT JOIN to distinguish race-not-found vs no-covering-jurisdiction in a single query

### Phase 7: Cron Automation + Auto-Upsert

**Goal**: Discovery runs automatically on a weekly schedule for all registered jurisdictions, and high-confidence candidates are upserted to race_candidates without requiring manual approval.

**Depends on**: Phase 6 (confidence scoring validated against real data via human review)
**Plans**: 2 plans

Plans:

- [x] 07-01: Auto-upsert branch + email-suppress flag in discoveryService.ts
- [x] 07-02: discoveryCron.ts (sweep + lock + sweep-summary email), thin cron wrapper, index.ts wiring, route 409 guards

**Details:**

- Auto-upsert eligibility: `raceId !== null && (confidence === 'official' || confidence === 'matched')` — uncertain and unmatched-race candidates always go pending
- SELECT-then-INSERT idempotency (not ON CONFLICT) — race_candidates has no unique index on (race_id, full_name)
- Auto-upserted candidates write a candidate_staging audit row: status='approved', reviewed_by='cron'
- suppressRunEmail only gates per-run review notification; zero-candidate regression + failure emails always fire
- In-process boolean lock (not Redis) — single-instance Render deployment; 2h TTL timer guards slow sweeps
- Routes use acquireRunLock() (atomic check-and-set) — prevents TOCTOU; isRunLockHeld exported as debugging aid only
- Lock released in .finally() on fire-and-forget promise chains — no leak on background run failure
- Sweep-summary email silence guard — suppressed when all outcome lists empty (no noise on clean weeks)
- Cron scheduled at '0 2 * * 0' (Sunday 02:00 UTC) — one hour before districtStaleness (03:00 UTC)

## Milestone Summary

**Key Decisions:**

- Citation-required agent output via forced `tool_choice=report_candidates` — 'any' lets Claude pick web_search as final tool; only forced tool ensures typed JSON
- NAME_MATCH_THRESHOLD = 0.85 — locked; 0.80 produced too many false-positive matches in testing
- No Postgres transaction wrapping orchestrator — run row IS the audit trail; partial failures preserved and visible
- confidence and flagged computed independently — confidence='official' + flagged=true = official source but no matching race in DB (ballot-completeness radar)
- Dual-router pattern: JWT-gated browser router + X-Admin-Token server-to-server router both under /api/admin; JWT mount first; auth at route level not mount level
- In-process boolean lock (not Redis) — correct for single-instance Render; process restart clears lock naturally
- Sequential for...of over jurisdictions — never Promise.all; exhausts Anthropic rate limit quota with no usable output
- Auto-upsert source label: 'discovery_cron' for cron path, 'discovery_admin' for manual approve path
- Discovery agent model: claude-opus-4-6 (Haiku doesn't support web_search_20250305; Opus too expensive was revised — actual cost ~$0.017/run with Sonnet; Opus used for quality)
- web_search max_uses: 1 when sourceUrl provided, 2 when not — prevents quota exhaustion

**Issues Resolved:**

- Mount-level requireAuth intercepted X-Admin-Token requests to sibling router — fixed by moving auth to route-level (f474d2a)
- STAG-04 gap: approve endpoints initially only updated staging status without promoting to race_candidates — fixed in 0d89b91 (found by audit)
- TypeScript `req.params` destructuring produces `string | string[]` — cast to `as string` consistent with all other routes in codebase

**Issues Deferred:**

- lavote.gov election ID (id query parameter) changes each cycle — mandatory manual update per election cycle, no automation
- Per-race agent scoping (current implementation runs full jurisdiction scan even when triggered by race ID)
- Multiple discovery_jurisdictions rows covering same (election_date, state) — at-most-one assumption documented

**Technical Debt Incurred:**

- Stale comment in stagingQueueAdmin.ts (cosmetic): JSDoc says auth applied at mount; actually per-route — no functional impact
- POST /discover/race/:id 202 happy path not directly tested in Phase 6 (no matching discovery_jurisdictions row existed at test time)
