<objective>
Close code-review finding CR-01 from Phase 210 (`.planning/phases/210-per-tab-compass-integration/210-REVIEW.md`):
the per-tab tab-entry effect can silently revert a lens the user just calibrated via the
"Calibrate" flow. Because `handleCalibrateLens` does a full-page navigation, `tabLensMemory`
is empty on return; CompassContext adopts the pending lens via `setActiveLens` and clears the
marker, but nothing records it into `tabLensMemory`, so a later async `rawUserAnswers`/`lenses`
update re-fires the Phase-210 tab-entry effect, which recomputes `resolveTabLens` against empty
memory → the tab default → clobbering the just-calibrated selection.

Fix: seed `tabLensMemory[activeView]` from the pending-calibration marker on the return mount,
treating the calibration as the tab's explicit pick (faithful to Phase 210 D-04). This is a
pure frontend state-wiring change — no backend, no new deps, no localStorage protocol change.
</objective>
