browseAreaRoute(candidate) -> "/results?browse_geo_id=...&browse_mtfcc=...&browse_label=...&from_locality=1"

<!-- COORDINATE HAND-OFF CONTRACT (SRCH-05, cross-page) — this is the single source of truth for the
     Landing -> Results coordinate navigation the checker required. Both Plan 03 (reader) and Plan 04
     (writer) depend on 214-02, so neither is a same-wave soft dependency on the other. -->
coordinateRoute(lat, lng, raw) -> "/results?lat=<lat>&lng=<lng>&coord_raw=<encodeURIComponent(raw)>"
  - lat, lng: the numeric pair produced by classifyInput (Plan 01).
  - raw: the LITERAL text the user typed (e.g. "39.17, -86.52"); carried so Results reconstructs the
    D-05 resting label from the user's own keystrokes, NEVER from the server response. These coordinates
    are client-sourced (user input placed in the user's own URL), so this preserves the Phase 213 /
    D-05 no-server-echo privacy contract.
  - URL-param names Results.jsx (Plan 03) MUST read on mount: `lat`, `lng`, `coord_raw`. When both `lat`
    and `lng` are present, Results resolves the coordinate through the SAME shared resolveCoordinate path
    its in-page onSubmitCoordinate uses (no duplicated logic).
