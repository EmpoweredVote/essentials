**CRITICAL HARD RULES — violation = fatal migration error:**

1. **NEVER include `slug` in any chamber INSERT.** `essentials.chambers.slug` is `GENERATED ALWAYS AS` — including it produces `ERROR: cannot insert into column "slug"`. The slug auto-derives from the chamber name.
2. **NEVER include `slug` in any government INSERT.** Same generated-column constraint applies if present (verify before writing — if slug exists on governments and is GENERATED, omit it).
3. **NO `ALTER TABLE essentials.chambers ADD COLUMN election_method`** — this column was added in Phase 41 migration 157 (Cambridge). It already exists. Adding it again will fail or no-op.
4. **NARROW district-table rule (CORRECTED from prior draft):** Do NOT insert TIGER-sourced district types (`STATE_UPPER`, `STATE_LOWER`, `NATIONAL_LOWER`, `COUNTY`) — those exist from Phase 49 TIGER load and must not be duplicated. **LOCAL-type district rows (`district_type='LOCAL'`, `mtfcc='G4110'`, `state='me'`) ARE REQUIRED for city address routing and MUST be inserted in this migration — one per city, 23 total.** Phase 49 loaded G4110 geofence_boundaries but created ZERO LOCAL district rows. `getRepresentativesByAddress` JOINs `office.district_id = district.id`, so any office with NULL district_id is silently dropped from results. Migration 167 (Cambridge) is the exact precedent — read it before writing any city block.
5. **NO `INSERT INTO essentials.election_races`** — election races are Phase 55 scope. Zero election_races rows in this file.
6. **NO new tables / NO schema changes.** Everything operates on the existing `essentials.governments`, `essentials.districts`, `essentials.chambers`, `essentials.offices` tables.
7. **`essentials.governments` has NO unique constraint on geo_id** (locked decision in STATE.md). Use `WHERE NOT EXISTS (SELECT 1 FROM essentials.governments WHERE geo_id = '...')` instead of `ON CONFLICT (geo_id)` — the latter will error. Same for `essentials.districts` (per Migration 167 line 19): use `WHERE NOT EXISTS` with `geo_id + district_type + mtfcc + state` matching.
8. **All 23 cities use `state='ME'` (uppercase) on `essentials.governments`** and `city=NULL` in the governments row — the city name is in the `name` field as `'City of {Name}, Maine, US'`. This matches the Cambridge/Phase 41 pattern. **HOWEVER, on `essentials.districts` rows, use `state='me'` (LOWERCASE)** — this matches the Phase 49 TIGER convention and Cambridge migration 167 was uppercase 'MA' but the explicit fix-hint for this revision specifies lowercase 'me' for ME LOCAL districts to align with the existing ME district rows from Phase 49. Verify against an existing ME district row before writing (`SELECT state FROM essentials.districts WHERE state ILIKE 'me' LIMIT 1`) and use the matching case. **The chosen case must be consistent across all 23 LOCAL district rows AND must match how Phase 49 stored existing ME district rows.**
9. **Every office row in this migration has `politician_id=NULL`, `is_vacant=true`, `is_active=true`.** These are skeletal rows awaiting incumbents.
10. **`representing_state='ME'` (uppercase) on every office row** — matches the Phase 51/52 ME pattern.
11. **`district_id` is initially set NULL in the office INSERTs, then back-filled by a per-city UPDATE step that sets it to the LOCAL district row's id** (see Step 3 in the city-block pattern below). At end of each city block, EVERY office row for that city must have `district_id` pointing to the city's LOCAL district row. This applies to BOTH council chambers and school chambers — Portland's school board offices ALSO point to the Portland LOCAL district row (not a sub-district, since no TIGER school-board layer exists).
12. **Wrap the ENTIRE migration in a single `BEGIN; ... COMMIT;`** — not per-city. If any city fails, the whole migration rolls back.
13. **Idempotency: every INSERT uses `WHERE NOT EXISTS`** — for governments by geo_id, for LOCAL districts by (geo_id, district_type, mtfcc, state), for chambers by (government_id, name), for offices by (chamber_id, title, seat_label) where seat_label is the district/ward/at-large identifier. The district_id back-fill UPDATE uses `WHERE district_id IS NULL` so re-runs are no-ops.
