## lacourt.org Search URLs
- Civil party-name search: https://www.lacourt.org/paos/v2public/CivilIndex/
- Criminal defendant search: https://www.lacourt.org/paos/v2public/CriminalIndex/
- Case summary (when case number is known): https://www.lacourt.ca.gov/casesummary/v2web3/
- Pricing: $1.00/search for registered users (1–10 searches/mo); $4.75 flat for guests.

## CRITICAL Search Mechanics (from 31-RESEARCH.md pitfall #1)
- PAOS is a **party/litigant name** index — it does NOT search by attorney of record.
- Search the firm name AS A PARTY: "Morrison & Foerster LLP" returns cases where MoFo itself is plaintiff or defendant.
- To find cases where the FIRM REPRESENTED a client before a specific judge, you must:
  1. Search the firm name; collect every case number that surfaces (firm-as-party or known associated cases).
  2. Open each case number in the case-summary URL.
  3. Read the case summary to identify (a) attorneys of record and (b) assigned judicial officer.
  4. ONLY count it as an appearance for a JUDGE candidate if the assigned judicial officer is that judge.
- For CITY ATTORNEY candidates (Ashouri, McKinney, Roy): appearances are recorded for any LA Superior Court case where the firm appears as party — no judge filter, no "before this judge" requirement.

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Manual lacourt.org research and results entry</name>
  <what-required>
Researcher (the user) performs lacourt.org searches for each firm in the workbook and edits `C:/EV-Accounts/backend/scripts/court-research-results.json` in place to record findings. This is unavoidably manual: lacourt.org PAOS has no API, charges $1.00–$4.75 per search, and ToS likely prohibits automated scraping.
  </what-required>
  <how-to-execute>
For each firm in the workbook (in the order printed):

1. **For judge candidates** (is_city_attorney=false):
   a. Go to https://www.lacourt.org/paos/v2public/CivilIndex/ — enter `raw_firm_name` as party name, search.
   b. Also try https://www.lacourt.org/paos/v2public/CriminalIndex/ — same firm name.
   c. For each case in results dated 2021-01-01 or later: click the case summary link.
   d. In the case summary, check the "Judicial Officer" / "Assigned Department" field.
   e. **Record the case ONLY IF** the assigned judicial officer matches the candidate judge's name.
   f. Cap at 10 cases per firm. If you would record an 11th, set `court_appearances_capped: true` and stop.
   g. Note any `RECUSAL`, `RECUSED`, `DISQUALIFICATION` text on the case summary — set `recusal_found: true` if any.

2. **For City Attorney candidates** (is_city_attorney=true):
   a. Same searches (Civil + Criminal).
   b. Record EVERY case dated 2021+ where the firm appears as a party — no judge filter required.
   c. Cap at 10 cases per firm.
   d. Recusal field is informational only for City Attorneys (does not change conflict logic, since CAs don't preside).

3. **Edit `C:/EV-Accounts/backend/scripts/court-research-results.json`** to fill the entry for that firm:
   - `court_appearances_found`: integer count of cases recorded (0 if none)
   - `court_appearances_capped`: true if you stopped at 10
   - `case_types`: array like `["Civil"]`, `["Civil","Criminal"]`, `["Family Law"]`
   - `first_appearance_date`, `last_appearance_date`: earliest and latest filing dates from recorded cases (null if zero cases)
   - `source_urls`: array of `https://www.lacourt.ca.gov/casesummary/v2web3/?caseNumber=...` links — one per recorded case
   - `recusal_found`: true if any recusal record observed
   - `conflict_note`: short free-text. Examples: `"Donor appeared, recusal filed in 23STCV12345"`, `null` (no special note)
   - `researcher_notes`: replace `"PENDING"` with what you searched + any oddities. Example: `"Searched 'MORRISON FOERSTER LLP' as Civil party; 3 cases 2022–2024 before Judge X; no recusal."`

4. **Zero-results entries**: leave `court_appearances_found: 0`, `source_urls: []`, dates `null`, but still set `researcher_notes` to something like `"Searched <variants>; no cases in 2021–present window."` so Plan 03 can distinguish "researched, none found" from "PENDING".

5. **needs_review firms**: for any firm flagged `needs_review: true` in the input, decide either:
   - It's the same firm as another nearby entry → research once, copy the result into both entries, prefix notes with `"merged with <other firm_name>; "`.
   - It's a distinct firm → research separately and prefix notes with `"distinct from <other firm_name>; "`.

6. **Re-validate JSON before signaling done**: open the file in your editor and confirm it parses (no trailing commas, all strings quoted). If unsure, `node -e "JSON.parse(require('fs').readFileSync('C:/EV-Accounts/backend/scripts/court-research-results.json','utf8')); console.log('OK')"`.

7. **Save and signal complete.**

If you choose to STOP early (only research a subset), that is acceptable: any entries left as `researcher_notes: "PENDING"` will be loaded with zero appearances. You can rerun Plan 02 + Plan 03 later to fill them in (Plan 03's UPSERT is idempotent).
