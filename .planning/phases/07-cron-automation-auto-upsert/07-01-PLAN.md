Extend runDiscoveryForJurisdiction() in discoveryService.ts to support an auto-upsert branch for high-confidence candidates and a per-run-email suppression flag.

Purpose: This is the foundation Plan 02 (the cron sweep) calls. Output: Modified discoveryService.ts with new opts handling, autoUpsert helper, and DiscoveryRunSummary.autoUpserted counter. No new files. No new packages.

Why this design (vs. alternatives):
- A single function with opts is simpler than a parallel runDiscoveryForJurisdictionWithAutoUpsert() (research Pattern 6 Option A).
- The audit row in candidate_staging keeps discovery_runs counts accurate and lets admin see the full history of every sweep, including auto-upserted items.
- SELECT-then-INSERT (instead of ON CONFLICT) is necessary because the table has no usable unique constraint for this case. Race condition is acceptable: two simultaneous runs never happen (Plan 02 enforces a global lock).
- last_verified_at is set on insert because the discovery agent literally just verified the candidate against allowed_domains.
- source='discovery_cron' makes auto-upserted rows distinguishable in the table from manual entries (sos_excel, county_clerk, manual).
