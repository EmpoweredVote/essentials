# Phase 97: Code Review Report

**Reviewed:** 2026-06-07
**Depth:** standard
**Files Reviewed:** 3
**Status:** issues_found

## Summary

Reviewed `gen_migration.py` (the generator script) and two generated SQL migration files covering 32 MD state senators across districts 16–47. The SQL structure is sound overall: `BEGIN`/`COMMIT` framing is correct, `ON CONFLICT (politician_id, topic_id) DO UPDATE` is present on both tables, dollar-quoting is used consistently (no `$REASON$` fallback was triggered, confirming no `$$` literals in any reasoning text), and URLs are single-quote-escaped correctly via `build_sources_array`.

One critical defect was found: every `sources` cast in both generated files uses `::text[]::text[]` — a redundant double cast that the generator emits because `build_sources_array` already appends `::text[]` and then `generate_migration` appends another `::text[]` at the call site. While Postgres silently accepts this, it signals the wrong return value from `build_sources_array` being blindly appended to by the caller.

Four warnings: (1) city-local topics (`local-environment`, `rent-regulation`, `residential-zoning`, `jail-capacity`) leak into migrations declared as "federal/state topics only" because `EXCLUDED_TOPICS_FEDERAL` is too narrow; (2) the name-keyed grouping in `generate_migration` is brittle — a name collision or CSV `full_name` mismatch silently produces zero stances for a candidate; (3) `float()` is used to validate the value but the float result is discarded and the raw row is appended, so a value like `"1.0  "` passes validation but the original whitespace-padded string ends up in the SQL; (4) the generator has no guard for the case where `read_csv_stances` returns rows for names not present in `candidate_inventory`, which means stances from wrong-named CSVs are silently dropped with no warning.

## Critical Issues

### CR-01: Double `::text[]` cast on every `sources` column (generated by `build_sources_array` + caller)

**File:** `C:/EV-Accounts/backend/data/stance-research/gen_migration.py:83,186`
**Also affects:** `C:/EV-Accounts/backend/migrations/284_md_senators_batch_b.sql` — all 258 `INSERT INTO inform.politician_context` statements (every single row)
**Also affects:** `C:/EV-Accounts/backend/migrations/285_md_senators_batch_c.sql` — all 220 `INSERT INTO inform.politician_context` statements

**Issue:** `build_sources_array` already returns strings ending in `::text[]` (line 83: `return f"ARRAY[{', '.join(sources)}]::text[]"` and line 82: `return "ARRAY[]::text[]"`). Then `generate_migration` (line 186) appends another `::text[]` literal:

```python
lines.append(f"        {sources_arr}::text[])")
```

This produces SQL like:

```sql
ARRAY['https://...']::text[]::text[]
```

on every row in both migration files (verified — 258 occurrences in 284, 220 in 285). PostgreSQL accepts the double cast without error because casting `text[]` to `text[]` is a no-op, but it is structurally wrong and will silently break if `build_sources_array` is ever changed to return a typed array with a different element type (e.g., `::varchar[]`), producing a cross-type cast instead of a no-op. It also makes the SQL visually misleading.

**Fix — two options (pick one):**

Option A: Remove the `::text[]` suffix from `build_sources_array`:
```python
def build_sources_array(row):
    sources = []
    for col in ['source_url_1', 'source_url_2', 'source_url_3']:
        url = (row.get(col) or '').strip()
        if url:
            url = url.replace("'", "''")
            sources.append(f"'{url}'")
    if not sources:
        return "ARRAY[]::text[]"   # keep type annotation on empty case
    return f"ARRAY[{', '.join(sources)}]"  # no cast; caller provides it
```

Option B: Remove the `::text[]` from the call site in `generate_migration` (line 186):
```python
lines.append(f"        {sources_arr})")  # sources_arr already carries ::text[]
```

Option B is lower-risk because the empty-array case `ARRAY[]::text[]` still needs the cast annotation for Postgres to know the element type, and `build_sources_array` already handles that correctly. Removing the duplicate from the call site (line 186) is the minimal fix.

### WR-01: City-local topics leak into "federal/state topics only" migrations

**File:** `C:/EV-Accounts/backend/data/stance-research/gen_migration.py:58`
**Also affects:** `C:/EV-Accounts/backend/migrations/284_md_senators_batch_b.sql:257-271, 2453-2483, 3621-3635, 3993+`
**Also affects:** `C:/EV-Accounts/backend/migrations/285_md_senators_batch_c.sql:1233-1248, 1773-1788, 3141-3157, 3401-3463`

**Issue:** The migration headers for 284 and 285 declare "Federal/state topics only; data-centers, local-immigration, transportation-priorities excluded." However `EXCLUDED_TOPICS_FEDERAL` only excludes those three keys:

```python
EXCLUDED_TOPICS_FEDERAL = {'data-centers', 'local-immigration', 'transportation-priorities'}
```

City-local topics that are in `TOPIC_UUIDS` but not in this exclusion set are passing through into the generated SQL. Confirmed present in the generated files:

- `local-environment` (UUID `1935979c-...`) — found for Jack Bailey, Arthur Ellis, Kevin M. Harris, Shaneka Henson, Benjamin F. Kramer, Bryan W. Simonaire, Sara Love, Alonzo T. Washington, and others in 284
- `rent-regulation` (UUID `c308e8e8-...`) — found for Sara Love and Jeff Waldstreicher in 284; Mary Washington in 285
- `residential-zoning` (UUID `d4f18138-...`) — found for Sara Love in 284
- `jail-capacity` (UUID `c267e137-...`) — found for Bill Ferguson, Antonio Hayes, Charles E. Sydnor III, Cory V. McCray, Mary Washington in 285

These topics were tagged as "City / local topics" in the `TOPIC_UUIDS` dict comment (line 41). Their presence in a state-senator migration is inconsistent with the stated scope. The practical risk: if the compass UI uses topic category to scope which politicians are shown for city-local questions, state senators with city-local stances will appear in or be scored against the wrong question set.

**Fix:** Either (a) extend `EXCLUDED_TOPICS_FEDERAL` to include all city-local topic keys, or (b) make the exclusion logic derive from the dict structure itself. Safest fix is explicit enumeration:

```python
EXCLUDED_TOPICS_FEDERAL = {
    'data-centers', 'local-immigration', 'transportation-priorities',
    # City-local topics — not applicable to state/federal officials:
    'city-sanitation', 'growth-and-development', 'jail-capacity',
    'judicial-access-to-justice', 'judicial-bail-pretrial',
    'judicial-government-deference', 'judicial-police-accountability',
    'judicial-prosecution-priorities', 'judicial-transparency',
    'local-environment', 'rent-regulation', 'residential-zoning',
}
```

Note: `judicial-criminal-justice` and `homelessness`/`homelessness-response` also appear in the city-local group in the dict comment and may warrant review.

### WR-02: Value float-validation discards the validated value; raw row string used in SQL

**File:** `C:/EV-Accounts/backend/data/stance-research/gen_migration.py:103-107`

**Issue:** In `read_csv_stances`, the value is parsed to float for validation (line 103-106), but if it passes, the raw row dict is appended (line 107). The row dict still contains the original raw string from the CSV. Then in `generate_migration` (lines 166-168), the value is parsed again from the raw row:

```python
try:
    value = float(row['value'].strip())
except ValueError:
    continue
```

This double-parse is redundant but also creates an inconsistency: if `row['value']` is something like `" 3 "` (whitespace-padded), the first parse in `read_csv_stances` succeeds, the row is kept, and then at generation time `float(" 3 ")` = `3.0` so the output SQL is correct. The real risk is that the float-to-string conversion in the generated SQL uses Python's default float repr (`3.0`, `5.0`), which emits `.0` — confirmed in both files. Postgres accepts `3.0` as a valid numeric literal so this is not a current bug, but the architecture is fragile. If someone stores the validated float in the row at parse time they also fix the redundancy.

**Fix:** Store the parsed float in `read_csv_stances` and use it downstream, removing the second parse:
```python
val = float(row.get('value', '').strip())
row['_value'] = val   # store parsed value in row
rows.append(row)
```
Then in `generate_migration`:
```python
value = row['_value']  # no second parse needed
```

### WR-03: Name-keyed grouping silently drops stances when CSV `full_name` mismatches inventory

**File:** `C:/EV-Accounts/backend/data/stance-research/gen_migration.py:116-122`

**Issue:** Stances are grouped by `full_name` from the CSV and then looked up by the same name from `candidate_inventory`. If a CSV uses a slightly different form of the name (e.g., `"William C. Smith Jr."` vs. `"William C. Smith, Jr."` in the inventory at line 429), the lookup `by_candidate.get(name, [])` returns an empty list, the block emits `-- NOTE: No stances found in CSV for {name}`, and all stances from that CSV are silently lost. There is no cross-check that confirms every name in `by_candidate` was actually consumed.

This is particularly risky because several names in the inventory contain punctuation that can vary in CSVs: `"William C. Smith, Jr."` (SD-20), `"Stephen S. Hershey, Jr."` (SD-36), `"Charles E. Sydnor, III"` (SD-44), `"C. Anthony Muse"` (SD-26), `"Jason C. Gallion"` (SD-35).

**Fix:** After the per-candidate loop, emit a warning for any name in `by_candidate` that was not found in `candidate_inventory`:
```python
inventory_names = {name for name, _ in candidate_inventory}
for csv_name in by_candidate:
    if csv_name not in inventory_names:
        print(f"  WARNING: CSV name '{csv_name}' not in candidate_inventory — stances dropped!", file=sys.stderr)
```

### WR-04: `build_sources_array` does not guard against URLs containing single quotes

**File:** `C:/EV-Accounts/backend/data/stance-research/gen_migration.py:79`

**Issue:** `build_sources_array` correctly escapes single quotes in URLs (`url.replace("'", "''")`, line 79), but does not guard against the URL containing a SQL comment sequence (`--`) or a dollar-sign sequence (`$$`). In practice, none of the current URLs in migrations 284/285 contain these sequences, so this is a latent risk rather than a current defect. However, the reasoning text in `dollar_quote` does have protection for `$$` (switching to `$REASON$` tag), and sources do not have analogous protection.

More concretely: if a URL ever contains `$$` (unlikely but possible in query-string parameters), the resulting SQL would be:

```sql
ARRAY['https://example.com?a=$$b']::text[]
```

which would be syntactically valid (single-quoted string) but visually alarming. The real SQL-breaking risk is `'` in a URL — which IS escaped. The `$$` in a URL inside a single-quoted string is not actually a SQL risk. This warning is primarily about the reasoning text, which goes through `dollar_quote`. However `reasoning` can currently contain backtick, `%`, and `_` without issue in dollar-quoted context.

**Fix:** Document that `build_sources_array` outputs single-quoted strings and `dollar_quote` outputs dollar-quoted strings, so `$$` in sources is not a SQL risk (the single-quote escaping is sufficient). The existing code is correct for this concern; add a comment making this explicit.

### IN-01: Redundant `::text[]` in the topic-UUID reference comment block header

**File:** `C:/EV-Accounts/backend/migrations/284_md_senators_batch_b.sql:14-57`
**Also affects:** `C:/EV-Accounts/backend/migrations/285_md_senators_batch_c.sql:14-57`

**Issue:** The header comment block lists all topics from `TOPIC_UUIDS` including city-local topics (`city-sanitation`, `growth-and-development`, `homelessness`, `homelessness-response`, `jail-capacity`, `judicial-*`, `local-environment`, `local-immigration`, `rent-regulation`, `residential-zoning`, `transportation-priorities`). These appear as reference comments even when the migration uses `EXCLUDED_TOPICS_FEDERAL`. This is not a bug, but a reader may assume all listed topics were actually inserted, which is misleading. The header already says "Federal/state topics only" but the full topic list makes it unclear which ones were actually applied.

**Fix:** Only list the topics that can actually appear in the migration body (i.e., filter the comment block by the active exclusion set), or add a note marking excluded topics with `(excluded)`.

### IN-02: `data-centers` is in `EXCLUDED_TOPICS_FEDERAL` but not in `TOPIC_UUIDS`

**File:** `C:/EV-Accounts/backend/data/stance-research/gen_migration.py:58,9-55`

**Issue:** `EXCLUDED_TOPICS_FEDERAL` includes `'data-centers'` (line 58) but `TOPIC_UUIDS` has no entry for `'data-centers'`. This means `data-centers` can never appear in any migration regardless of the exclusion set — it would be caught by the `if topic_key not in TOPIC_UUIDS` check on line 98 with a WARNING first. The exclusion is therefore dead code. Similarly `EXCLUDED_TOPICS_LOCAL` also includes `'data-centers'`. This is benign but adds confusion — if a future CSV contains a `data-centers` row, the script will emit an unknown-topic warning rather than silently filtering it via the exclusion set.

**Fix:** Either add `data-centers` to `TOPIC_UUIDS` with its UUID (if it is a real topic in the DB), or remove it from both exclusion sets and rely on the existing unknown-topic warning to catch stray CSVs.
