<objective>
Add the per-race discovery trigger endpoint (SCHED-02): `POST /api/admin/discover/race/:id`. This lets an admin kick off a discovery run for a single race without knowing the jurisdiction UUID. The endpoint resolves the covering jurisdiction by JOINing `races → elections → discovery_jurisdictions` on `(election_date, state)`, then delegates to the existing `runDiscoveryForJurisdiction` orchestrator (which still scopes the run to the whole jurisdiction — discovery is not actually single-race at the agent level, but the API surface is race-scoped for UI convenience).

Purpose: SCHED-02 from the roadmap. The use case is admin-in-UI pressing "Re-run discovery for this race" on a race detail page (future Phase 7 work may add that UI; this plan only adds the backend endpoint so the trigger is available).

Output: One additional handler in the existing `essentialsDiscovery.ts` router. No new files, no index.ts changes, no new dependencies. The router is already mounted under `/api/admin` with `requireAdminToken` from Phase 5.

Scope limits (out of scope for this plan):
- No `GET /admin/discover/races` listing endpoint — the research document suggests it but CONTEXT.md does not require it, and the UI consumer is a future concern.
- No per-race scope limiting inside `runDiscoveryForJurisdiction` — the run still covers the whole jurisdiction (all races for that election_date + state). That's a Phase 7 concern if needed.
- No duplicate-run locking — Phase 7 will add a run lock when cron is introduced. For on-demand re-triggers of the same jurisdiction, duplicate runs will currently happen (both produce discovery_runs rows; both stage). Acceptable for Phase 6.
</objective>

Insert the following handler:

```typescript
// ---------------------------------------------------------------------------
// POST /discover/race/:id
// Auth: requireAdminToken (applied at mount in index.ts)
// Resolves the discovery_jurisdictions row that covers this race by JOINing
// races → elections → discovery_jurisdictions on (election_date, state), then
// delegates to runDiscoveryForJurisdiction. There is no race_id column on
// discovery_jurisdictions — discovery is jurisdiction-scoped at the agent level.
// Returns 202 immediately; run continues in background.
// ---------------------------------------------------------------------------
router.post('/discover/race/:id', async (req: Request, res: Response): Promise<void> => {
  try {
    const raceId = req.params.id as string;

    if (!raceId || !UUID_REGEX.test(raceId)) {
      res.status(422).json({ code: 'VALIDATION_ERROR', message: 'Invalid race id (expected UUID)' });
      return;
    }

    // Resolve the covering discovery_jurisdictions row.
    // A race maps to a jurisdiction via the election's (election_date, state) pair — the
    // same pair used when seeding discovery_jurisdictions in Phase 5.
    const resolveResult = await pool.query<{
      race_exists: boolean;
      jurisdiction_id: string | null;
      jurisdiction_name: string | null;
    }>(
      `WITH race_row AS (
         SELECT r.id AS race_id, r.election_id, e.election_date, e.state
           FROM essentials.races r
           JOIN essentials.elections e ON e.id = r.election_id
          WHERE r.id = $1
       )
       SELECT
         (SELECT true FROM race_row) AS race_exists,
         dj.id   AS jurisdiction_id,
         dj.jurisdiction_name
         FROM race_row rr
         LEFT JOIN essentials.discovery_jurisdictions dj
           ON dj.election_date = rr.election_date AND dj.state = rr.state`,
      [raceId]
    );

    if (resolveResult.rows.length === 0) {
      // The CTE returned no rows → the race itself doesn't exist (no join rows either)
      res.status(404).json({ code: 'RACE_NOT_FOUND', message: 'Race not found' });
      return;
    }

    const row = resolveResult.rows[0];
    if (!row.jurisdiction_id) {
      res.status(404).json({
        code: 'NO_DISCOVERY_JURISDICTION',
        message: 'No discovery_jurisdictions row covers this race. Register a jurisdiction with matching election_date + state first.',
      });
      return;
    }

    const jurisdictionId = row.jurisdiction_id;
    const jurisdictionName = row.jurisdiction_name;

    // Fire-and-forget; run row persists status='running' immediately, so callers can poll.
    runDiscoveryForJurisdiction(jurisdictionId, { triggeredBy: 'on_demand' }).catch((err) => {
      console.error('[discoverRace] background run failed for raceId=' + raceId + ' jurisdictionId=' + jurisdictionId + ':', err);
    });

    res.status(202).json({
      status: 'accepted',
      raceId,
      jurisdictionId,
      jurisdictionName,
    });
  } catch (err) {
    console.error('[POST /discover/race/:id] error:', err);
    res.status(500).json({ code: 'INTERNAL_ERROR', message: 'An unexpected error occurred' });
  }
});
```

**Why the CTE pattern for race resolution**: A naive `JOIN` approach returns zero rows whether the race doesn't exist OR the jurisdiction isn't registered — we can't distinguish the two cases for proper 404 codes. The CTE + LEFT JOIN preserves the race row even when no jurisdiction matches, so we can detect "race exists but no jurisdiction covers it" vs "race doesn't exist at all". The outer SELECT returns ≥1 row if and only if the race exists; within that row, `jurisdiction_id IS NULL` means no covering jurisdiction.

**Edge case — multiple covering jurisdictions**: In theory a race could have multiple `discovery_jurisdictions` rows covering its (election_date, state) combination (e.g. county-level and city-level jurisdictions for the same date). The LEFT JOIN will return multiple rows in that case. The handler uses `.rows[0]` — i.e. it picks the first one arbitrarily. This is acceptable because per CONTEXT.md the jurisdiction registration model is 1:1 per (election_date, geoid), and geoid is scoped to jurisdiction_level. If this limitation becomes real, address it in Phase 7 with explicit jurisdiction_level filtering. For now: document the assumption in a comment.

Add an inline comment above the query:

