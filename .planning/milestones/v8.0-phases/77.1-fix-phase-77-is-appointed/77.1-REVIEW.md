## Warnings

### WR-01: Silent zero-row UPDATE masks data gap if source rows are missing

**File:** `C:/EV-Accounts/backend/migrations/235_fix_portland_officials_is_appointed.sql:21-23`

**Issue:** The UPDATE statement has no enforcement that exactly 2 rows were modified. If `external_id` values `-690003` or `-690004` are absent from `essentials.politicians` (e.g., if migration 231 was rolled back, or if the external_id scheme changed), the UPDATE returns 0 rows affected without raising an error. The ledger INSERT on line 25 still executes successfully, permanently recording version `'235'` as applied. A future operator checking the ledger would see it as complete, with no indication that the intended data fix never landed. The affected officials would remain with incorrect `is_appointed=false` values with no audit trail of the failure.

**Fix:** Add a row-count assertion after the UPDATE using a `DO` block or a `RAISE` in a PL/pgSQL anonymous block. The minimal pattern used in several migrations in this codebase is:

```sql
DO $$
DECLARE
  updated_count INT;
BEGIN
  UPDATE essentials.politicians
  SET is_appointed = true
  WHERE external_id IN (-690003, -690004);

  GET DIAGNOSTICS updated_count = ROW_COUNT;

  IF updated_count <> 2 THEN
    RAISE EXCEPTION 'Migration 235: expected 2 rows updated, got %', updated_count;
  END IF;
END $$;
```

This causes the transaction to roll back if the UPDATE does not touch exactly the 2 expected rows, preventing a misleading ledger entry.

## Info

### IN-01: Header comment says "Applied: 2026-05-30" before migration has been applied

**File:** `C:/EV-Accounts/backend/migrations/235_fix_portland_officials_is_appointed.sql:2`

**Issue:** The comment on line 2 reads `-- Applied: 2026-05-30`. This date is written pre-emptively before the migration has actually been executed against the database. It also conflates the authored date with the applied date — if application is delayed past 2026-05-30, the header will be incorrect. Other migrations in this codebase use the same convention, so this is a systemic pattern rather than a unique defect here, but it is still factually inaccurate at review time.

**Fix:** Either change the label to `-- Authored: 2026-05-30` (and separately record the applied date when known), or leave the field blank and fill it in after successful application. The operational risk is low — it is documentation only.
