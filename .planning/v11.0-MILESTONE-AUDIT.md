## Integration Issues

### BLOCKERS

**1. Migration number collision — Plan 90-03 (POST-ELECTION-01/02)**

`90-03-PLAN.md` targets migration `268` and STATE.md line 58 reads `Next migration: 268`. However:
- `268_finance_summary_column.sql` was already applied to production
- Phase 92 consumed migrations 269, 270, 271

The correct next migration number is **272**. Plan 90-03 must be updated from `268` to `272` before execution, and STATE.md must be corrected. If run as-is, the migration will either collide with the existing 268 ledger entry or silently create a duplicate.

**Files to fix:**
- `.planning/STATE.md` line 58: `Next migration: 268` → `Next migration: 272`
- `.planning/phases/90-post-election-follow-up-minicompass-ui/90-03-PLAN.md`: all `268` references → `272`

### WARNINGS

**2. MD-GOV-04 / Phase 93 planning gap — 141 delegates → 71 polygon mapping**

MD has 141 house delegates across 47 multi-member districts. Phase 91 loaded 71 SLDL polygon rows (one per letter sub-district, e.g., 47A and 47B). Phase 93 must map each delegate to the correct sub-district polygon. This mapping strategy (which sub-district polygon each delegate resides in) has not been documented. Phase 93 CONTEXT must specify this before planning begins — otherwise the seed migration will fail or produce incorrect district linkages.

**3. MiniCompass.jsx ringColor prop silently dropped (UI-01/UI-02)**

`MiniCompass.jsx:164` passes `ringColor="transparent"` to RadarChartCore, but this prop does not exist in ev-ui 0.9.4. Guide rings remain visible in the compact overlay. The human verifier approved the current state, so this is not a functional regression — but if ring suppression is desired, a `showRings` prop would need to be added to ev-ui in a future release.

**4. STATE.md progress metrics inconsistent (INFO)**

STATE.md frontmatter: `completed_phases: 1`, `percent: 10`. Actual state: 2 full phases complete (91, 92) + Phase 90 at 2/3 plans. This is a documentation stale issue that will confuse Phase 93 planning if not corrected.
