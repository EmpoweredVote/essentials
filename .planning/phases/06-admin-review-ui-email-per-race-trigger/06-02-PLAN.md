<objective>
Add email notifications to the discovery pipeline (OBS-02, OBS-03):

1. **Review notification** — when a discovery run produces uncertain candidates, send a summary email to the admin with counts by confidence tier and a link to the review page. Mark urgent in the subject when the election is within 30 days.
2. **Failure alert** — when a discovery run throws and ends in the `failed` branch, send an email with the error message.
3. **Zero-candidate regression alert** — when a run completes with zero candidates but the previous completed run for the same jurisdiction had non-zero candidates, send an email flagging the regression.

All three fire from inside `runDiscoveryForJurisdiction` in `lib/discoveryService.ts`. The `emailService.sendEmail` function already exists and never throws (catches its own errors), so these calls are safe to `await` without wrapping in try/catch.

Purpose: OBS-02 ("Admin receives email when uncertain candidates are queued for review — urgency-aware") + OBS-03 ("Admin receives email when a discovery run fails or returns unexpectedly zero results"). Without these emails the admin has no passive signal that the review queue needs attention; they'd have to check the UI on a schedule.

Output: A single modified file (`discoveryService.ts`) with:
- Three new counters (`uncertainStaged`, `matchedStaged`, `officialStaged`) populated inside the existing staging loop
- A success-branch email block after step 7 (review notification + zero-candidate regression alert)
- A failure-branch email block in step 8 before `throw err`
- An extended `DiscoveryRunSummary` type exposing the three counters
- A small `buildReviewEmailHtml` helper function for the review notification body

No new files, no new dependencies. The existing `sendEmail` (raw-fetch Resend) is used as-is.

```typescript
/**
 * Compose the review-notification email body.
 * Plain HTML — no templating engine. Styling kept minimal so it renders consistently
 * across email clients.
 */
function buildReviewEmailHtml(args: {
  jurisdictionName: string;
  uncertainStaged: number;
  matchedStaged: number;
  officialStaged: number;
  withdrawalsStaged: number;
  reviewUrl: string;
  daysUntilElection: number | null;
}): string {
  const urgencyLine =
    args.daysUntilElection !== null && args.daysUntilElection <= 30
      ? `<p><strong>Election is ${args.daysUntilElection} day(s) away.</strong></p>`
      : '';
  return `
    <div style="font-family: system-ui, sans-serif; max-width: 560px;">
      <h2 style="margin: 0 0 8px 0;">Candidates need review — ${args.jurisdictionName}</h2>
      ${urgencyLine}
      <ul style="line-height: 1.6;">
        <li><strong>${args.uncertainStaged}</strong> uncertain</li>
        <li><strong>${args.matchedStaged}</strong> matched</li>
        <li><strong>${args.officialStaged}</strong> official</li>
        ${args.withdrawalsStaged > 0 ? `<li><strong>${args.withdrawalsStaged}</strong> possible withdrawal(s)</li>` : ''}
      </ul>
      <p>
        <a href="${args.reviewUrl}"
           style="display:inline-block;padding:10px 16px;background:#1f6feb;color:#fff;border-radius:6px;text-decoration:none;">
          Review queue
        </a>
      </p>
    </div>
  `;
}
```

```typescript
    const adminEmail = process.env.ADMIN_EMAIL;
    const reviewUrl = process.env.ADMIN_REVIEW_URL ?? 'https://essentials.empowered.vote/admin/staging';

    if (adminEmail) {
      const daysUntilElection = Math.ceil((cfg.election_date.getTime() - Date.now()) / 86400000);

      // Review notification — fires when any candidates were staged
      if (candidatesStaged > 0) {
        const isUrgent = daysUntilElection <= 30;
        const subject = isUrgent
          ? `[URGENT] ${uncertainStaged} candidates need review — ${cfg.jurisdiction_name} election in ${daysUntilElection} days`
          : `${uncertainStaged} candidates need review — ${cfg.jurisdiction_name}`;

          jurisdictionName: cfg.jurisdiction_name,
          uncertainStaged,
          matchedStaged,
          officialStaged,
          withdrawalsStaged,
          reviewUrl,
          daysUntilElection: isUrgent ? daysUntilElection : null,
        });
        await sendEmail({ to: adminEmail, subject, html });
      }

      // Zero-candidate regression alert — fires when this run returned zero but a previous run did not
      if (agentResult.candidates.length === 0) {
        const prevResult = await pool.query<{ candidates_found: number }>(
          `SELECT candidates_found
             FROM essentials.discovery_runs
            WHERE discovery_jurisdiction_id = $1
              AND status = 'completed'
              AND id <> $2
            ORDER BY completed_at DESC NULLS LAST
            LIMIT 1`,
          [cfg.id, runId]
        );
        const prevCount = prevResult.rows[0]?.candidates_found ?? null;
        if (prevCount !== null && prevCount > 0) {
          await sendEmail({
            to: adminEmail,
            subject: `Zero candidates returned — ${cfg.jurisdiction_name} (was ${prevCount})`,

                <h2 style="margin: 0 0 8px 0;">Zero candidates returned</h2>
                <p>Jurisdiction: <strong>${cfg.jurisdiction_name}</strong></p>
                <p>This run found 0 candidates. The previous completed run found <strong>${prevCount}</strong>.</p>
                <p>This may indicate an upstream source change (HTML restructure, URL moved, etc.). Investigate the allowed_domains source pages.</p>
              </div>
            `,
          });
        }
      }
    }
```

**Why this placement is correct**:
- After step 7's UPDATE: the run is already marked `completed`. A sendEmail failure cannot cause double-finalization.
- Before `return`: the Promise returned by `runDiscoveryForJurisdiction` completes only after emails are attempted. This is correct for on-demand runs (admin waits for 202 ack from the trigger endpoint, which already returns immediately). For cron, the sequential runner will wait — which is fine since sendEmail is fast.
- `sendEmail` never throws (it logs and returns). Safe to `await` without try/catch.
