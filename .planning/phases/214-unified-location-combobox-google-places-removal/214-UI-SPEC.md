# Phase 214 — UI Design Contract

> Visual and interaction contract for the shared Location Combobox (Results header + Landing search bar) and the full retirement of Google Places. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — **shadcn NOT initialized.** `components.json` absent; the codebase (24 milestones, 213 prior phases) already has an established hand-rolled Tailwind v4 + CSS-custom-property design system (`src/index.css` `@theme` tokens + `:root` `--ev-*` vars) used consistently across every existing component (`LocalityMatches.jsx`, `Results.jsx`, `ResultsHeader.jsx`, etc.). Introducing shadcn for one component would fork the design system rather than extend it — this phase deliberately reuses the existing system instead. **Flag for the human operator:** if you actually want shadcn initialized project-wide, say so explicitly before planning starts; this spec assumes "no" by default per the existing precedent. |
| Component library | none (hand-rolled JSX + Tailwind utility classes; no Radix/base-ui) |
| Icon library | none — hand-authored inline SVG, Feather-icon style: `viewBox="0 0 24 24"`, `fill="none"`, `stroke="currentColor"`, `strokeWidth="2"`, `strokeLinecap="round"`, `strokeLinejoin="round"`. New icons for this phase (loading spinner, Enter-hint glyph) MUST match this exact style — do not import a new icon package. |
| Font | `--font-sans: "Inter"` (default UI/body text, incl. the combobox `<input>`) and `--font-display: "Manrope"` (reserved for the resolved location label + election-summary text, matching the existing `formattedAddress` span pattern at `Results.jsx:1896`) |
| Positioning primitive | `@floating-ui/react` (already a dependency) — use it for the combobox listbox's anchor/positioning instead of hand-rolled `absolute` offsets |

## Spacing Scale

Declared values (must be multiples of 4), matching existing usage throughout `Results.jsx` / `LocalityMatches.jsx`:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gap inside a listbox row (`gap-1`), badge internal padding |
| sm | 8px | Gap between pin icon and input text; gap between badges (`gap-2`) |
| md | 16px | Input horizontal padding (`px-4`); listbox row horizontal padding (`px-4`) |
| lg | 24px | Combobox-to-adjacent-content gaps; header horizontal padding on mobile (`px-6`) |
| xl | 32px | Header horizontal padding on desktop (`sm:px-12`, already 48 — see exception) |
| 2xl | 48px | Header desktop horizontal padding (existing `sm:px-12`) |

Exceptions:
- **44×44px minimum touch target** on every interactive control in the combobox (submit button, listbox option rows, any clear/dismiss affordance) — this is the existing app-wide convention (`min-w-[48px] min-h-[44px]` on the current Search button, `w-11 h-11` = 44px on the edit-pencil button). Non-negotiable for the new component; do not shrink below this on mobile.
- Listbox row vertical padding stays `py-3` (12px, matches existing `LocalityMatches` rows) — not on the 8-pt scale but is the established precedent; do not "fix" it to 16px, it would break visual parity with rows that predate this phase (none remain, but keep 12px for row density).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body (input text, resolved location label) | 16px | 400 (regular) | 1.5 |
| Label (state qualifier, hint/error inline copy, "Elections" summary) | 14px | 400 (regular) | 1.4 |
| Micro-badge (`kind` pill, "Stances" badge, coordinate-hint glyph) | 12px | 600 (semibold) | 1.2 |
| Emphasis (matched place-name in listbox row, resting location label) | 16px | 600 (semibold) | 1.5 |

Only 2 weights in play: **400 regular** (body/hint/error copy) and **600 semibold** (emphasis + badges). The pre-existing Search button used `font-bold` (700) — **do not carry that third weight into the new combobox**; restyle its label to 600 semibold for consistency with this contract (visually indistinguishable at button size, keeps the type system honest at 2 weights).

Heading/Display roles: **not applicable this phase** — the combobox introduces no new headings; existing page-level headings elsewhere are untouched.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--ev-bg-light` `#f0f8fa` (light) / `--color-ev-navy` `#0d1117` (dark) | Page background behind the combobox row |
| Secondary (30%) | `#ffffff` (light) / `--color-ev-navy-card` `#161b22` (dark) | Input background, listbox dropdown surface, inline hint/error row background |
| Accent (10%) | `--ev-teal` `#00657c` (light) / `--color-ev-teal-light` `#00c8d7` (dark, text-only) | **Reserved for exactly:** the location pin icon, the submit/"Search" button background (`#00657c` both modes — button keeps solid teal even in dark mode since text is white-on-teal), the focus ring (`focus:ring-2 focus:ring-[var(--ev-teal)]`), the matched place-name text in listbox rows, the active/hovered listbox row background tint (`--ev-bg-light` light / `gray-800` dark — unchanged from existing `LocalityMatches` pattern), and the "Stances" coverage badge |
| Error / invalid input | `--ev-coral` `#ff5740` | **Reserved for exactly:** the 3 coordinate 422 messages (`INVALID_COORDINATES` / `OUTSIDE_US_BOUNDS` / `SWAPPED_COORDINATES`) and the zero-candidate "no matches" inline row's accent icon/text. This is an error/invalid-state color, not a destructive-action color — **Phase 214 has no destructive actions** (no delete/remove flows), so there is no confirmation-dialog use of coral here. |

Accent reserved for: location pin icon · submit button background · input focus ring · matched-label text in listbox rows · active listbox row highlight · "Stances" coverage badge. Never used for plain body text, borders on non-active elements, or decorative fills.

| Element | Copy |
|---------|------|

Unchanged from the current submit button label — kept for continuity across the redesign; the button now submits whichever path the D-02 classifier picked (address / coordinate), not just address. |
| Placeholder (empty field, no location set yet) | `"Address, city, or coordinates"` |
| `aria-label` on the input | `"Search by address, city, county, state, or decimal coordinates"` |
| Enter-hint row (D-01, address-shaped input) | `"Press Enter to look up this address"` — small inline row below the field, 14px regular, gray-500/gray-400 dark, preceded by a small "↵" glyph (12px semibold, same style family as the micro-badge tokens) |
| Enter-hint row (coordinate-shaped input — Claude's discretion default, consistent with D-01's intent) | `"Press Enter to look up this location"` |
| Empty state | **Not applicable on focus** — D-04 locks "nothing on empty focus, no dropdown until typed." There is no empty-state panel to write copy for. |
| Zero-candidate / no-match state (name-like query, resolver returns 0 candidates) | Heading-less inline row: `"No matches for “{query}.” Check the spelling, or press Enter to search it as a street address."` — 14px regular, coral-accented leading icon (a small "!" or "x" glyph, 12px) |
| Error — `INVALID_COORDINATES` | `"That doesn't look like a valid coordinate. Use decimal degrees, like 39.17, -86.52."` |
| Error — `OUTSIDE_US_BOUNDS` | `"Those coordinates are outside the United States. Enter a US location."` |
| Error — `SWAPPED_COORDINATES` | `"Latitude and longitude look swapped — try lat, lng, like 39.17, -86.52."` |
| Resolved candidate row — area-type tag | Existing `kind` values, unchanged casing/style from `LocalityMatches.jsx` (`city` / `county` / `state`) |
| Resolved candidate row — coverage badge | `"Stances"` — shown only when the `/location-search` response's `has_local_data` flag is true (Phase 212 D-05/D-07 contract) |
| Destructive confirmation | **None — Phase 214 introduces no destructive actions.** (No delete/remove/clear-with-confirm flow; the pre-filled input's focus-selects-all behavior is the only "replace" affordance, and it is non-destructive/undo-free by nature of a text field.) |

## Combobox Interaction States (visual reference for the executor)

This maps CONTEXT.md's locked decisions (D-01 through D-09) onto concrete visual states. Read `.planning/phases/214-unified-location-combobox-google-places-removal/214-CONTEXT.md` for the full rationale — this table is the quick-glance visual contract.

| State | Trigger | Visual treatment |
|-------|---------|-------------------|
| Resting | Page load / after a resolve | Single `<input>`, pre-filled with the resolved label (address via `toAddressTitleCase`, place via `City, ST`, or the raw typed `lat, lng` per D-05 — never server-echoed). Pin icon left (16px, teal/teal-light dark). Submit button right. No dropdown, no hint row. |
| Focused (no typing yet) | `onFocus` | Text auto-selected (select-all) so the next keystroke replaces it cleanly. Still no dropdown (D-04) — focus alone never opens the listbox. |
| Typing — name-like | Regex fails address/coordinate heuristics (D-02) | After ~250ms debounce and ≥3 chars: a subtle 16px teal spinner appears briefly right-inside the input (left of the submit button), then the listbox opens below the field showing ranked candidates (reusing `LocalityMatches` row markup: bold teal label + `, ST` qualifier + `kind` pill + conditional "Stances" badge). `role="listbox"`/`role="option"`, `aria-activedescendant` tracks arrow-key highlight. |
| Typing — address-like | Leading digits / street-number pattern (D-02) | No dropdown. Inline hint row appears below the field (14px, gray, "↵ Press Enter to look up this address"). No resolver call fires. |
| Typing — coordinate-like | `lat, lng` decimal regex matches (D-02) | No dropdown, no resolver call. Inline hint row shows the coordinate variant copy. |
| Zero candidates | Name-like query, `/location-search` returns `[]` | Listbox closes; the no-match inline row (coral-accented icon, see Copywriting Contract) replaces the hint row position. |
| Candidate selected | Click/Enter on a listbox row | Listbox closes, input value updates to the candidate's resolved label, navigation dispatches to the existing `coverageAreaToPath`/browse route (unchanged downstream). |
| Coordinate submitted | Enter/Search-click on a coordinate-shaped value | `POST /essentials/coordinate-lookup` fires; on success the resting label becomes the client-side-reconstructed `lat, lng` text (D-05 — never from the server response); on a 422, the matching coral error row renders in place of the hint row, and the field stays in edit state (no navigation, no results swap). |
| Address submitted | Enter/Search-click on an address-shaped value | Existing Census geocode path (`handleAddressSearch`) fires unchanged; on success resting label = `toAddressTitleCase(formattedAddress)`. |
| Dark mode | `.dark` class on `<html>` | Every state above themes via existing Tailwind `dark:` variants + the `--color-ev-navy*`/`--color-ev-teal-light` tokens already defined in `src/index.css`'s `@theme` block — no new tokens needed, no `!important` required (this is a new component, not an `ev-ui`-package-owned one). |

**Input attributes** (accessibility + anti-regression): `type="text"` (never `type="search"`/`"tel"` — keeps the mobile keyboard alphabetic), `autoComplete="off"`, `spellCheck={false}`, `role="combobox"`, `aria-expanded`, `aria-controls` pointing at the listbox `id`, `aria-autocomplete="list"`. No dedicated "clear" (×) button — the focus-select-all behavior (D-03) is the only clear affordance, keeping the field a single stateless input as CONTEXT.md's D-03 explicitly requires (no pill↔input state machine).

**Deletions verified against this spec** (D-09, end-of-phase grep gate): `useGooglePlacesAutocomplete.js`, the `.pac-container`/`.pac-item*` CSS block in `src/index.css`, the Google `classifyQuery` path in `src/lib/localitySearch.js`, and `@googlemaps/js-api-loader` in `package.json` must all be gone with zero `google`/`pac-container`/`window.google` hits outside deleted files.
