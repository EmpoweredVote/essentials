  <files>(no file changes — DB read + HTTP POST)</files>
  <action>
Look up Plano's `discovery_jurisdictions.id` (UUID, created by 16-01) and trigger a manual discovery run via the admin endpoint.

Steps:
1. Source backend env: `set -a; source /c/EV-Accounts/backend/.env; set +a`
2. Get Plano's discovery_jurisdictions.id:
   ```bash
   PLANO_DJ_ID=$(psql "$DATABASE_URL" -t -A -c "SELECT id FROM essentials.discovery_jurisdictions WHERE state='TX' AND jurisdiction_name='Plano' ORDER BY election_date DESC LIMIT 1;")
   echo "Plano discovery_jurisdictions.id = $PLANO_DJ_ID"
   ```
3. Confirm `$PLANO_DJ_ID` looks like a UUID. If empty: STOP — 16-01 did not seed Plano correctly.
4. Read `ADMIN_TOKEN` from backend env (already loaded in step 1 — confirm with `echo "${ADMIN_TOKEN:0:6}..."` to check it's set without leaking).
5. Confirm the live backend URL — the deployed Render URL for accounts-api. Per memory: deploys via push to master at C:\EV-Accounts. Confirm by:
   ```bash
   curl -s -o /dev/null -w "%{http_code}\n" https://accounts-api.onrender.com/health
   ```
   Expected: 200. If non-200, fall back to local `http://localhost:8080` (only if backend is running locally — else surface as blocker).
6. Note the start timestamp BEFORE triggering (for filtering discovery_runs in Task 2):
   ```bash
   RUN_START=$(date -u +%Y-%m-%dT%H:%M:%SZ)
   echo "Run start (UTC): $RUN_START"
   ```
7. Trigger the discovery run (note: `essentialsDiscovery` router is mounted at `/api/admin` in `src/index.ts`):
   ```bash
   curl -i -X POST "https://accounts-api.onrender.com/api/admin/discover/jurisdiction/$PLANO_DJ_ID" \
     -H "X-Admin-Token: $ADMIN_TOKEN" \
     -H "Content-Type: application/json"
   ```
8. Capture the HTTP status and response body. The discovery agent runs synchronously OR queues — either is acceptable. Expected status: 200 or 202.

Important: This call may take 30-90 seconds (the discovery agent uses claude-sonnet-4-6 to crawl the source). Allow it to complete. If curl times out before getting a response, the run may still be progressing — Task 2 will check the DB directly.

If the response is 401 or 403: ADMIN_TOKEN mismatch or backend env not loaded. STOP and report.
If the response is 404: route not mounted or the UUID is wrong. STOP and report.
If the response is 5xx: capture the response body — likely an agent error. Report as gap.
  </action>
  <verify>
- `$PLANO_DJ_ID` matches UUID format
- `curl https://accounts-api.onrender.com/health` returns 200
- POST to `/api/admin/discover/jurisdiction/$PLANO_DJ_ID` returns 200 or 202 (not 4xx/5xx)
- Response body either describes a completed run (with candidates_found > 0) OR confirms run was started
  </verify>

  <name>Task 2: Verify discovery_runs and candidate_staging in DB</name>
  <files>(no file changes — DB queries)</files>
  <action>
Confirm the discovery run reached the database and produced at least one staged candidate with a valid citation_url.

Steps (use the `$PLANO_DJ_ID` and `$RUN_START` captured in Task 1):

1. Verify a discovery_runs row exists for this trigger:
   ```bash
   psql "$DATABASE_URL" -c "
     SELECT id, discovery_jurisdiction_id, status, started_at, completed_at, candidates_found
     FROM essentials.discovery_runs
     WHERE discovery_jurisdiction_id = '$PLANO_DJ_ID'
       AND started_at >= '$RUN_START'::timestamptz
     ORDER BY started_at DESC
     LIMIT 1;
   "
   ```
   Expected: exactly one row. Capture the `id` as `$RUN_ID`. Note `status` (success/error/in_progress) and `candidates_found`.

   If zero rows: the run never reached the DB — the route may not be wired or the service didn't write a run row. Report as blocker.
   If status is `error`: capture the error message column (likely `error` or `error_message`) and report. Note: a single failed run still proves the WIRING works; the failure may be a domain-specific blocker (e.g., source URL unreachable). Report findings to user via the human-verify checkpoint in Task 3.
   If status is `in_progress` after 3 minutes: poll once more then report whatever state the row reaches.

2. Count staged candidates from this run:
   ```bash
   psql "$DATABASE_URL" -c "
     SELECT COUNT(*) FROM essentials.candidate_staging
     WHERE run_id = '$RUN_ID';
   "
   ```

   Expected: >= 1 (per DISC-03 success criterion). If 0: the agent ran but found no candidates. Document this — it is acceptable per CONTEXT.md ("source_url is set now even for cities with no active races") but DISC-03 explicitly requires Plano's test to produce >= 1 candidate. Surface this to the user in the Task 3 checkpoint with the agent's reasoning if available.

3. Verify every staged candidate has citation_url within Plano's allowed_domains:
   ```bash
   psql "$DATABASE_URL" -c "
     WITH plano_domains AS (
       SELECT unnest(allowed_domains) AS domain
       FROM essentials.discovery_jurisdictions
       WHERE id = '$PLANO_DJ_ID'
     )
     SELECT cs.id, cs.full_name, cs.citation_url
     FROM essentials.candidate_staging cs
     WHERE cs.run_id = '$RUN_ID'
       AND NOT EXISTS (
         SELECT 1 FROM plano_domains pd
         WHERE cs.citation_url ILIKE '%' || pd.domain || '%'
       );
   "
   ```
   Expected: zero rows (every staged candidate's citation_url contains one of Plano's allowed_domains).

   If any rows return: domain whitelist enforcement is not working — report as gap in Task 3 checkpoint.

4. Sample staged candidates for the human-verify checkpoint:
   ```bash
   psql "$DATABASE_URL" -c "
     SELECT full_name, race_hint, citation_url, status
     FROM essentials.candidate_staging
     WHERE run_id = '$RUN_ID'
     ORDER BY created_at
     LIMIT 5;
   "
   ```
   Capture the output for display in Task 3.
