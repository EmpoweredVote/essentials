<task type="checkpoint:human-verify" gate="blocking">
  <what-built>
Phase 8 deliverable: a single admin page at `/admin/discovery` with three sections (Jurisdictions, Run History, Coverage Health), JWT-gated, that lets admins trigger and monitor candidate discovery runs entirely from the browser.

Backend additions:
- Migration 083 adds `candidates_auto_upserted` to `essentials.discovery_runs`
- `discoveryService.ts` now persists that count on run completion
- New JWT-gated endpoints: `GET /api/admin/discovery/jurisdictions`, `GET /api/admin/discovery/runs`, `GET /api/admin/discovery/coverage`

Frontend additions:
- `src/pages/admin/DiscoveryDashboard.jsx` (the page)
- Four helpers in `src/lib/adminApi.js`
- Route `/admin/discovery` in `App.jsx`
  </what-built>
  <how-to-verify>
1. **Start the stack locally** (or visit a deployed preview environment if both backend and frontend are deployed). Confirm the backend is on the expected port/URL and the frontend dev server is running.

2. **Log in as an admin user**, then navigate to `/admin/discovery`.

3. **Section 1 — Jurisdictions table** (ADMUI-01):
   - Confirm there is one dense row per `discovery_jurisdictions` entry.
   - Columns visible: Jurisdiction, Election (date), Status badge, Last Run (timestamp), Last Added, Total Active, Run Discovery button.
   - Click each column header — rows resort. Click again — sort reverses.
   - Type a partial jurisdiction name into the search box — list filters live.
   - For a jurisdiction that has never been run, the status badge reads "Never run" (gray).

4. **Section 1 — Run Discovery flow** (ADMUI-02 + ADMUI-03):
   - Pick a jurisdiction with status Success or Never run. Click its **Run Discovery** button.
   - **Immediately** (before polling): the status badge flips to a yellow pulsing "Running" pill, AND the button is replaced by a teal spinner.
   - Wait. The page polls every ~4 seconds. When the run completes:
     - The badge transitions to Success (green) or Failed (red).
     - The Last Run timestamp updates.
     - Last Added shows the new staged count.
     - The Run Discovery button reappears.
     - A toast appears in the top-right (green for success showing the staged count; red for failure showing a brief reason).
   - Try clicking Run Discovery TWICE in rapid succession on the same row. The second click should not produce a duplicate run; you should see either a 409 toast ("Run already in progress") or no second run in the history.

5. **Section 2 — Run History** (DASH-01 + DASH-02):
   - Confirm the most recent runs (including the one you just triggered) appear at the top.
   - Each row shows: Jurisdiction, When (date+time), Found, Staged, Auto-Upserted, Status badge, Trigger ("Cron" or "Manual").
   - Pagination: if there are more than 25 runs, click Next — older runs load. Click Prev — return.
   - Filter by jurisdiction via the dropdown — table narrows to that jurisdiction's runs only.
   - Confirm the **Auto-Upserted** column shows non-zero values for any cron run that auto-upserted candidates (this validates Plan 08-01 persisted the column).

6. **Section 3 — Coverage Health** (DASH-03):
   - Confirm one row per jurisdiction.
   - Columns: Jurisdiction, Election, Total Races, With Candidates, Zero Candidates.
   - The "Zero Candidates" cell is **red and bold** when > 0; gray when 0.
   - After your test run completes, confirm this section auto-refreshed in step with the jurisdictions section (same poll cycle).

7. **Auth gate**: open `/admin/discovery` in a private/incognito window. You should be redirected to login (RequireAuth) — no data is visible without admin auth.

8. **No new package dependencies**: `git diff package.json` shows no changes (or only changes you expected from another concurrent task).

If any step fails, describe what went wrong, which jurisdiction/run was involved, and any console / network-tab errors observed.
