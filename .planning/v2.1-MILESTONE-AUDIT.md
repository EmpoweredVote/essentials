## Critical Gap: STAG-04 — Manual Approve Does Not Promote to race_candidates

### What the requirement says
> "Admin can approve a staged candidate via API endpoint (triggers upsert to race_candidates)"

### What was built
Two approve endpoints exist:
- `POST /api/admin/discovery/staging/:id/approve` (JWT, browser UI) — `stagingQueueAdmin.ts` lines 92-100
- `POST /api/admin/discovery/staging/:id/approve` (X-Admin-Token, server-to-server) — `essentialsDiscovery.ts` lines 202-210

Both endpoints only execute:
```sql
UPDATE essentials.candidate_staging SET status='approved', reviewed_at=now(), reviewed_by=... WHERE id=$1 AND status='pending'
```

Neither calls `autoUpsertToRaceCandidates()`. The function exists in `discoveryService.ts` but is private (non-exported) and is only invoked from `runDiscoveryForJurisdiction()` when `opts.autoUpsert === true` — which the cron sweep passes but manual trigger paths do not.

### Impact
When an admin:
1. Triggers a manual discovery run (on-demand)
2. Sees uncertain/flagged candidates in the staging queue
3. Reviews and clicks Approve

The candidate is marked `approved` in `candidate_staging` but is **never written to `race_candidates`**. It remains invisible to voters until the next Sunday cron sweep re-discovers the same candidate as high-confidence and auto-upserts it — which may not happen if confidence scoring changes.

### Fix Required
Export `autoUpsertToRaceCandidates` from `discoveryService.ts` and call it from both approve handlers when `race_id IS NOT NULL AND action != 'withdrawal'`. Alternatively, add an inline upsert equivalent in the approve handler.

**Files to change:**
- `C:\EV-Accounts\backend\src\lib\discoveryService.ts` — export `autoUpsertToRaceCandidates`
- `C:\EV-Accounts\backend\src\routes\stagingQueueAdmin.ts` — call after status UPDATE, return result in response
- `C:\EV-Accounts\backend\src\routes\essentialsDiscovery.ts` — same fix for X-Admin-Token approve endpoint

## E2E Flow Status

### Flow 1: Cron Sweep Auto-Upsert ✓ COMPLETE
```
discoverySweep.ts Sunday 02:00 UTC
→ acquireRunLock (409 if already held)
→ runDiscoverySweep()
→ for each jurisdiction (sequential)
→ runDiscoveryForJurisdiction({ autoUpsert: true, suppressRunEmail: true })
→ agent finds candidates
→ high-confidence (official/matched) + raceId not null
  → autoUpsertToRaceCandidates() → race_candidates
  → candidate_staging audit row (status='approved', reviewed_by='cron')
→ uncertain → candidate_staging (status='pending')
→ sweep-summary email (when outcomes non-empty)
→ releaseRunLock()
```
Status: **COMPLETE**

### Flow 2: Manual Trigger → Stage → Admin Review ✓ COMPLETE (partial — see Flow 3)
```
POST /discover/jurisdiction/:id (X-Admin-Token)
→ acquireRunLock (409 if held)
→ 202 Accepted
→ fire-and-forget: runDiscoveryForJurisdiction({ triggeredBy: 'on_demand' })
→ candidates → candidate_staging (status='pending')
→ per-run review email (when candidatesStaged > 0)
→ admin opens /admin/staging in browser
→ StagingQueue.jsx loads pending rows via GET /api/admin/discovery/staging
→ admin sees candidate with confidence badge, citation URL, race context
→ admin clicks Approve or Dismiss
```
Status: **COMPLETE through dismiss; Approve breaks before race_candidates (see Flow 3)**

### Flow 3: Manual Approve → race_candidates ✗ BROKEN

Admin clicks Approve in StagingQueue.jsx
