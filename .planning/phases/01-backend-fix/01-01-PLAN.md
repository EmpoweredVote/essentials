  Make exactly these changes to `getElectionsByCoordinate()` in electionService.ts. The research doc (01-RESEARCH.md) has exact code examples for each change.

  **1. ElectionRow interface (around line 37-56):** Mark candidate columns as nullable. Change:
  - `candidate_id: string` -> `candidate_id: string | null`
  - `full_name: string` -> `full_name: string | null`
  - `is_incumbent: boolean` -> `is_incumbent: boolean | null`
  - `candidate_status: string` -> `candidate_status: string | null`
  (first_name, last_name, photo_url, politician_id are already nullable — no change needed)

  **2. Part A geofence query (around line 186):** Change `JOIN essentials.race_candidates rc ON rc.race_id = r.id` to:
  ```sql
  LEFT JOIN essentials.race_candidates rc
    ON rc.race_id = r.id
    AND rc.candidate_status != 'withdrawn'
  ```
  CRITICAL: Remove `AND rc.candidate_status != 'withdrawn'` from the WHERE clause (around line 200). If left in WHERE, it turns the LEFT JOIN back into an effective INNER JOIN because NULL != 'withdrawn' evaluates to NULL (falsy). This is the #1 pitfall.

  **3. Part B statewide query (around line 252):** Same change — replace `JOIN essentials.race_candidates rc ON rc.race_id = r.id` with:
  ```sql
  LEFT JOIN essentials.race_candidates rc
    ON rc.race_id = r.id
    AND rc.candidate_status != 'withdrawn'
  ```
  And remove `AND rc.candidate_status != 'withdrawn'` from that query's WHERE clause as well.

  **4. Deduplication filter (around line 270-276):** Wrap the dedup set operations in a null guard so that rows with NULL candidate_id (0-candidate races) always pass through. Change from:
  ```typescript
  const dedupedRows = allRows.filter((row) => {
    if (seenCandidates.has(row.candidate_id)) return false;
    seenCandidates.add(row.candidate_id);
    return true;
  });
  ```
  To:
  ```typescript
  const dedupedRows = allRows.filter((row) => {
    if (row.candidate_id !== null) {
      if (seenCandidates.has(row.candidate_id)) return false;
      seenCandidates.add(row.candidate_id);
    }
    return true;
  });
  ```
  Without this, all 0-candidate race rows collapse into one because Set treats repeated null additions as duplicate.

  **5. Grouping loop candidate push (around line 316-326):** Wrap the candidate construction and push in a null guard. Change from:
  ```typescript
  const candidate: ElectionCandidate = {
    candidate_id: row.candidate_id,
    ...
  };
  racesMap.get(row.race_id)!.candidates.push(candidate);
  ```
  To:
  ```typescript
  if (row.candidate_id !== null) {
    const candidate: ElectionCandidate = {
      candidate_id: row.candidate_id,
      full_name: row.full_name!,
      first_name: row.first_name,
      last_name: row.last_name,
      photo_url: row.photo_url,
      is_incumbent: row.is_incumbent!,
      candidate_status: row.candidate_status!,
      politician_id: row.politician_id,
    };
    racesMap.get(row.race_id)!.candidates.push(candidate);
  }
  ```
  Note the non-null assertions (`!`) on `full_name`, `is_incumbent`, `candidate_status` — these are safe inside the `candidate_id !== null` guard because those columns are always populated when a candidate row exists.

  **Do NOT change:**
  - The `getCandidateById()` function — it queries a single candidate by ID; no LEFT JOIN needed there.
  - The `inferDistrictType()` function — unrelated utility.
  - The `ElectionCandidate`, `ElectionRace`, or `ElectionResult` interfaces — these describe the output shape, which is unchanged (races get an empty array, not null fields).
  - The `LEFT JOIN LATERAL` on `politician_images` — it is already a LEFT JOIN and works correctly when `rc.politician_id` is NULL.
  - The ORDER BY clauses — NULLs sort last with DESC (PostgreSQL default), which is correct.
  - The early-exit guard (`if (dedupedRows.length === 0) return []`) — still correct; if no rows at all, no elections matched.
