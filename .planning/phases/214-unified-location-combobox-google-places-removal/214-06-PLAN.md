<task type="checkpoint:human-verify" gate="blocking">
  <name>Task 1: Verify the shared combobox on Results + Landing</name>
  <what-built>
    A single shared `<LocationCombobox>` now powers both the Results header and the Landing search bar, replacing the Address/Browse toggle + LocationBrowser tree and Landing's Google-Places-bound input. It classifies typed input (address / place name / decimal coordinates) client-side and dispatches to the Census address path, the Phase 212 /location-search resolver, or the Phase 213 /coordinate-lookup endpoint. Google Places, its hook, LocalityMatches, LocationBrowser, the .pac CSS, and the @googlemaps dependency are all removed.
  </what-built>
  <how-to-verify>
    Run `npm run dev` and open the app. On the RESULTS page:
    1. Place name: type "Springfield" — after ~250ms a dropdown of ranked candidates appears, each showing a `City, ST` / `County, ST` / `ST` qualifier and a "Stances" badge on covered areas (SRCH-02/04). No dropdown appears before 3 characters.
    2. Keyboard only: with the dropdown open, press ArrowDown/ArrowUp to move the highlight (focus stays in the input), press Enter to select — it navigates to that location (SRCH-02). Press Escape to dismiss.
    3. Ambiguity: confirm a same-named place (e.g. "Springfield") lists multiple state-qualified rows — never auto-picks one (SRCH-04).
    4. Address: type a full street address (e.g. "1600 Pennsylvania Ave, Washington, DC") — NO dropdown; an inline "↵ Press Enter to look up this address" hint shows; press Enter → officials resolve via the Census path; resting label is the title-cased address (SRCH-03).
    5. Coordinates: type "39.17, -86.52" — inline coordinate hint shows; press Enter → officials for that point render; the field's resting label reads exactly "39.17, -86.52" (the typed text, NOT a server address), and the location banner is correct/plausible for that point, not a neighboring city (SRCH-05, D-05, banner guard).
    6. Coordinate error: type an out-of-US or swapped pair (e.g. "51.5, -0.12" London, or "-86.52, 39.17" swapped) — an inline coral error message appears, the field does NOT navigate and does NOT clear.
    7. Empty focus: click the pre-filled field — text selects-all and NO dropdown opens (D-03/D-04).
    On the LANDING page: repeat steps 1, 4, and 5 in the search bar and confirm identical behavior (same component — SRCH-06); confirm the coverage list below still renders and browses (D-04), and the separate "search candidates by name" box still works.
    Dark mode: toggle dark mode and confirm the input, dropdown, hint row, and coral error row all theme correctly with no unreadable contrast.
  </how-to-verify>
  <resume-signal>Type "approved" if all checks pass, or describe the specific failures (which page, which input type, which step) so a gap-closure plan can target them.</resume-signal>
</task>
