<objective>
Thread the backend `locality` field (shipped live in 216-02) through the essentials frontend and render
"Unincorporated {County}, ST" for points that fall outside any incorporated place in a place-loaded
state — for BOTH address searches and coordinate searches.

Purpose: Fix the misleading banner where an unincorporated parcel silently displays its nearest postal
city (e.g. "Tucson") as if it were the governing municipality. Backend already emits the authoritative
signal; this plan renders it (LOC-04).
Output: Correct locality banner on Results for unincorporated points; unchanged behavior everywhere else.

**Repo:** this repo (essentials), Render on push to `main`. Depends on 216-02 (locality live in prod)
so the developer can test against the real field.

## Artifacts this phase produces
- **`unincorporatedLabel(localityStatus)`** — new pure helper in `src/lib/localityLabel.js`:
  returns `"Unincorporated " + county_name` when `incorporated === false && county_name`, else `null`.
- **`coordLocality`** — new `useState(null)` in Results.jsx, the coordinate-mode parallel to the
  hook's address-mode locality (mirrors how `browseResults` is a second data channel alongside the hook).
- **`locality`** field on searchPoliticians()/lookupCoordinate() returns + usePoliticianData return.

## Critical implementation note (RESEARCH Pitfall 1 — the #1 risk)
`usePoliticianData` (and its tribalLand/soon-locality state) is wired for ADDRESS mode ONLY.
`Results.jsx` derives list/phase from `browseResults` when `searchMode==='coordinate'`, and the hook's
`enabled` flag is false for coordinate searches (activeQuery is cleared). Mirroring tribal_land's
hook-based wiring ALONE will silently NOT render the label for coordinate searches. The coordinate path
MUST populate a separate `coordLocality` state inside `resolveCoordinate()`. Both modes must be exercised.
</objective>
