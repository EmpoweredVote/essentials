<objective>
Add a backend `locality` probe to the shared core `resolveOfficialsAtPoint` in accounts-api,
mirroring the `tribal_land` precedent exactly: two dedicated narrow `ST_Covers` geofence probes
(place `mtfcc IN ('G4110','G4120')`, county `mtfcc = 'G4020'`) run in the existing `Promise.all`,
gated to place-loaded states, and attached to the response at both return points. Expose `locality`
on the `/candidates/search` subset object; the coordinate route inherits it verbatim.

Purpose: The frontend cannot distinguish a postal city from an incorporated place (D-01) — only the
backend has the geofence geometry. This plan produces the authoritative `locality` signal the
frontend (216-03) renders as "Unincorporated {County}, ST".
Output: `locality` field live in the accounts-api response shape (LOC-01, LOC-02, LOC-03).

**Repo:** C:/EV-Accounts (accounts-api backend). Commit via `git -C "C:/EV-Accounts"` per project
memory — this repo pushes `master` → Render. Do NOT commit these backend files into the essentials repo.

## Artifacts this phase produces (register for downstream consumers)
- **`locality` object shape** (new field on `AddressSearchResult`, returned by both routes):
  `locality: { incorporated: boolean | null, place_name: string | null, county_name: string | null }`
  - place-loaded state + G4110/G4120 hit → `{ incorporated: true,  place_name: "<name>", county_name }`
  - place-loaded state + no place hit    → `{ incorporated: false, place_name: null,     county_name }`
  - un-loaded state (or no state)        → `{ incorporated: null,  place_name: null,     county_name }`
- **`buildLocality(state, placeRows, countyRow)`** — new exported pure helper in essentialsService.ts (gate logic; unit-tested).
- **`placeQueryText`, `countyNameQueryText`** — new SQL probe constants in resolveOfficialsAtPoint.
- **`PLACE_LOADED_STATES`** — new `Set<string>` const = `['AZ','CA','IN','ME','MD','MA','NV','OR','TX','UT','VA']` (11 states; MO excluded).
- Frontend symbols (produced in 216-03): `unincorporatedLabel()` helper, `coordLocality` state, `locality`/`incorporationInfo` threading.

## Planner decisions (explicit — do not treat as silent side effects)
- **PLACE_LOADED_STATES is a hardcoded 11-state list, NOT a dynamic "any G4110 row" gate.**
  Rationale: the ⭐ authoritative DB ground-truth query in 216-CONTEXT.md (run 2026-07-22) shows
  Missouri has 1 incidental G4110 place — a naive dynamic gate would wrongly include MO and
  false-positive real MO cities as "unincorporated". A threshold-based dynamic gate adds a
  per-request query + a magic constant with no readability gain. A static list also matches the TIGER
  loader's own "adding a state is a code change, on purpose" philosophy. The 11-state list supersedes
  RESEARCH.md's 10-state static analysis (which missed IN=566 places — IN IS loaded per live DB).
- **The pre-existing `tribal_land` coordinate-mode wiring gap (RESEARCH Open-Q2) is DEFERRED,
  not fixed here.** Keeping Phase 216 scoped to LOC-01..04. The tribal gap is pre-existing,
  independent of this feature, and would add parallel badge-render surface with its own test needs.
  Track it as a separate follow-up. This plan does NOT touch tribal_land wiring.
