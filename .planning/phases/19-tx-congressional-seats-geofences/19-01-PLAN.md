```sql
-- Migration 104: Backfill district_id on TX NATIONAL_LOWER districts.
--
-- The load-us-congressional-boundaries.ts script populates geo_id/ocd_id/label/
-- district_type/state/mtfcc but leaves district_id NULL. Migration 054-style
-- election linking (and the politician migration in 19-03) expect district_id
-- to be the un-padded numeric string ('3' for TX-3, '23' for TX-23, etc.).
--
-- This UPDATE strips the leading '48' state FIPS prefix and then strips any
-- leading zeros from the district number portion.
--
-- Idempotent: only updates rows where district_id IS NULL or empty.

UPDATE essentials.districts
SET district_id = LTRIM(SUBSTRING(geo_id FROM 3), '0')
WHERE state = 'TX'
  AND district_type = 'NATIONAL_LOWER'
  AND (district_id IS NULL OR district_id = '');
```

Rules:
- Use `LTRIM(SUBSTRING(geo_id FROM 3), '0')` so that '4803' → '3', '4823' → '23', '4838' → '38'.
- Scope strictly to `state = 'TX' AND district_type = 'NATIONAL_LOWER'` — do not touch any other district rows.
- The `(district_id IS NULL OR district_id = '')` guard makes this idempotent and safe to re-run.

Then apply it:
1. Source env: `set -a; source /c/EV-Accounts/backend/.env; set +a`
2. Apply: `psql "$DATABASE_URL" -f /c/EV-Accounts/backend/migrations/104_tx_congressional_district_id_backfill.sql`
3. Output should show `UPDATE 38` (or fewer if a partial backfill happened earlier — every additional run shows `UPDATE 0`).

Why this matters: The Plan 19-03 politician migration looks up districts by geo_id (which is fine), but downstream election linking (e.g. when TX congressional elections are eventually seeded) uses `WHERE d.district_id = parsed.dist_num`. Without this backfill, those JOINs return zero rows.
