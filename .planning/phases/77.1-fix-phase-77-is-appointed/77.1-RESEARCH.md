# Phase 77.1: Fix Phase 77 data — set politicians.is_appointed=true for Lee III and Taylor

**Researched:** 2026-05-30
**Domain:** PostgreSQL data fix — essentials.politicians UPDATE migration
**Confidence:** HIGH

## Summary

Phase 77 (migration 231) correctly set `offices.is_appointed_position=true` for Raymond C. Lee III (City Administrator, external_id=-690003) and Robert L. Taylor (City Attorney, external_id=-690004), but left `politicians.is_appointed=false` on both rows. The plan spec acknowledged this explicitly: "is_appointed=false (column on politicians table — leave false; the office row carries the appointed flag)." That decision was wrong — both columns should reflect appointment status.

The `politicians.is_appointed` field is a separate boolean that the backend reads and surfaces via the API (in `essentialsService.ts` and `essentialsBrowseService.ts`). It is returned as `is_appointed` in representative responses alongside `is_elected` (derived from `offices.is_appointed_position`). Setting it correctly matters for profile display and any downstream logic that consults `p.is_appointed` directly.

The fix is a single-statement SQL migration: `UPDATE essentials.politicians SET is_appointed = true WHERE external_id IN (-690003, -690004)`. This is the minimal, surgical change needed — no offices rows change, no cascade risk, no schema change.

**Primary recommendation:** Write migration 235 as a two-row UPDATE scoped to `external_id IN (-690003, -690004)`, wrapped in `BEGIN; … COMMIT;`, with a ledger entry for version '235'.

### Pattern 1: Single-table UPDATE fix migration (with ledger)

This is the established pattern for surgical data fixes in this codebase. See `223a_or_executive_district_fix.sql` as the closest analog — a minimal `UPDATE` wrapped in `BEGIN; … COMMIT;` with an idempotency-scoped `WHERE` clause.

```sql
-- Source: 223a_or_executive_district_fix.sql (project codebase)
BEGIN;

UPDATE essentials.politicians
SET is_appointed = true
WHERE external_id IN (-690003, -690004);

INSERT INTO supabase_migrations.schema_migrations (version)
VALUES ('235')
ON CONFLICT (version) DO NOTHING;

COMMIT;
```

**Idempotency:** Re-running this UPDATE when `is_appointed` is already `true` is a no-op (UPDATE returns 0 rows changed). The `ON CONFLICT DO NOTHING` on the ledger entry is also idempotent.

### Anti-Patterns to Avoid

- **Modifying offices rows:** `offices.is_appointed_position` is already correct (`true` for both). Do NOT touch offices in this migration.
- **Using politician UUID directly:** Always match on `external_id` — UUIDs are non-deterministic and harder to audit. External IDs -690003 and -690004 are confirmed from migration 231 and the summary.
- **Missing ledger entry:** Every applied migration needs a version entry in `supabase_migrations.schema_migrations` for ledger integrity.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Idempotent UPDATE | Custom pre-check logic | WHERE clause scoped to target rows | Postgres UPDATE on already-correct value is a no-op; no guard needed |
| Migration versioning | Custom version tracking | `supabase_migrations.schema_migrations` ON CONFLICT DO NOTHING | Established pattern across all migrations in this codebase |

## Common Pitfalls

### Pitfall 1: Updating offices instead of politicians
**What goes wrong:** Confusing `offices.is_appointed_position` (already correct) with `politicians.is_appointed` (the column being fixed).
**Why it happens:** Both columns have similar names; migration 231's comments focus on `is_appointed_position`.
**How to avoid:** The UPDATE targets `essentials.politicians`, not `essentials.offices`. Pre-flight query confirms current state: `SELECT external_id, is_appointed FROM essentials.politicians WHERE external_id IN (-690003, -690004)` — expected to return `false, false` before the fix.
**Warning signs:** If the UPDATE returns 0 rows affected, either the WHERE clause is wrong or the fix already ran (check ledger).

### Pitfall 2: Wrong migration number
**What goes wrong:** Using 233 or 234 (already occupied), or skipping to 236.
**Why it happens:** Migration list not re-verified at execution time.
**How to avoid:** Confirm via directory listing that 234 is the last file; use 235.
**Warning signs:** Ledger INSERT conflict on a version that shouldn't exist.

### Pitfall 3: Forgetting the ledger entry
**What goes wrong:** Migration applied but not recorded in `supabase_migrations.schema_migrations`, breaking the ledger sequence.
**Why it happens:** Simple one-liner migrations feel like they don't need bookkeeping.
**How to avoid:** Every migration in this codebase has a ledger entry. Always include it.
